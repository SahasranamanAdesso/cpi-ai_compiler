## Executive Summary

Version 1.1 Phase 1 introduces an **AI Frontend** that generates TypeScript code using the SAP Integration SDK from natural language descriptions.

**Key Achievement:** The AI generates ONLY TypeScript. The compiler (v1.0) remains completely unchanged.

---

## Architecture

### Complete Flow

```
Natural Language Request
        ↓
   PromptBuilder
  (Build system prompt)
        ↓
   AIProvider
  (Send to LLM)
        ↓
   SDKValidator
  (Validate TypeScript)
        ↓
Generated TypeScript Code
  (using SAP Integration SDK)
        ↓
   [Manual Step - Phase 2 will automate this]
        ↓
   Compiler v1.0
   (UNCHANGED)
        ↓
   HelloWorld.zip
```

### Separation of Concerns

The AI Frontend is **completely independent** of the compiler:

```
┌─────────────────────────────────────────────────────────┐
│                    AI Frontend (v1.1)                   │
│  Natural Language → TypeScript                          │
│                                                          │
│  - PromptBuilder                                        │
│  - AIProvider (Claude, OpenAI, etc.)                    │
│  - SDKValidator                                         │
│  - AIPipeline                                           │
└─────────────────────────────────────────────────────────┘
                         │
                         │ Generated TypeScript
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│                 Compiler v1.0 (FROZEN)                  │
│  TypeScript → XML/BPMN/ZIP                              │
│                                                          │
│  - Model                                                │

```

## Components Implemented

### 1. AIProvider Interface

**File:** `src/ai/AIProvider.ts`

```typescript
export interface AIProvider {
    generate(prompt: string): Promise<string>;
}
```

**Purpose:**
- Abstract interface for AI/LLM providers
- Allows pluggable providers (Claude, OpenAI, Gemini, Ollama)
- Compiler never depends on specific provider

**Responsibility:**
- Send prompt to LLM
- Return raw response

**Non-Responsibility:**
- Validation
- Compilation
- Prompt building

### 4. SDKValidator

**File:** `src/ai/SDKValidator.ts`

```typescript
export class SDKValidator {
    validate(code: string): ValidationResult
}
```

**Validation Checks:**
- ✓ Code is not empty
- ✓ Contains exactly one IFlow
- ✓ Contains at least one Component
- ✓ Uses only supported SDK classes (IFlow, Component, Connection)
- ✓ Uses only supported component types (Enricher)
- ✓ Exports the flow
- ✓ No XML/BPMN generation
- ✓ No file/ZIP generation
- ✓ No Markdown code blocks

**Validation Result:**
```typescript
interface ValidationResult {
    isValid: boolean;
    errors: string[];
}
```

**Design Decision:**
- Lightweight validation only
- No TypeScript compilation
- No semantic analysis
- Keep v1.1 simple

### 5. AIPipeline

**File:** `src/ai/AIPipeline.ts`

```typescript
export class AIPipeline {
    constructor(
        provider: AIProvider,
        promptBuilder?: PromptBuilder,
        validator?: SDKValidator
    )
    
    async generate(userRequest: string): Promise<GenerationResult>
    async generateWithRetry(userRequest: string, maxRetries?: number): Promise<GenerationResult>
}
```

**Orchestration Workflow:**
1. Build prompt from user request
2. Send prompt to AI provider
3. Validate generated TypeScript
4. Return result

**Features:**
- Dependency injection (pluggable components)
- Automatic retry with error feedback
- Provider-agnostic
- Clean error handling

**Generation Result:**
```typescript
interface GenerationResult {
    success: boolean;
    code: string;
    errors: string[];
    validationResult?: ValidationResult;
}
```

## SOLID Principles Applied

### Single Responsibility Principle (SRP)
- **PromptBuilder** - Only builds prompts
- **AIProvider** - Only calls LLM API
- **SDKValidator** - Only validates TypeScript
- **AIPipeline** - Only orchestrates workflow

### Open/Closed Principle (OCP)
- New providers can be added without modifying AIPipeline
- Extensible through interfaces

### Liskov Substitution Principle (LSP)
- Any AIProvider implementation can be used interchangeably

### Interface Segregation Principle (ISP)
- AIProvider interface has single method (minimal)
- No fat interfaces

### Dependency Inversion Principle (DIP)
- AIPipeline depends on AIProvider interface, not concrete implementations
- Dependency injection for all components

## Usage Example

### Basic Usage

```typescript
import { AIPipeline } from './src/ai/AIPipeline';
import { ClaudeProvider } from './src/ai/providers/ClaudeProvider';

// 1. Create provider
const provider = new ClaudeProvider(process.env.ANTHROPIC_API_KEY!);

// 2. Create pipeline
const pipeline = new AIPipeline(provider);

// 3. Generate TypeScript from natural language
const result = await pipeline.generate(
    "Create a flow that receives HTTPS requests and sets the message body to 'Hello World'"
);

// 4. Check result
if (result.success) {
    console.log('Generated TypeScript:');
    console.log(result.code);
    // Code = valid TypeScript using SAP Integration SDK
} else {
    console.error('Errors:', result.errors);
}
```

### With Retry

```typescript
// Retry up to 3 times if validation fails
const result = await pipeline.generateWithRetry(
    "Create a multi-step Integration Flow",
    3
);
```

### Custom Provider

```typescript
class OpenAIProvider implements AIProvider {
    async generate(prompt: string): Promise<string> {
        // OpenAI implementation
    }
}

const pipeline = new AIPipeline(new OpenAIProvider());
```

## Design Decisions

### Why Lightweight Validation?

**Decision:** Basic syntax checks only (no TypeScript compilation)

**Rationale:**
- Keep v1.1 simple
- Faster validation
- Less dependencies
- Compiler will catch deeper issues anyway

**Future:** v1.2 can add semantic validation if needed

---

### Why Provider Interface?

**Decision:** Abstract AI providers behind interface

**Rationale:**
- Compiler never depends on specific LLM
- Easy to add OpenAI, Gemini, Ollama
- Users can bring their own provider
- Testing easier (mock providers)

**Future:** Add more providers in v1.2

---

### Why Separate AI Frontend?

**Decision:** AI Frontend is independent of compiler

**Rationale:**
- Compiler v1.0 is proven and frozen
- AI can evolve independently
- Clear separation of concerns
- Easy to test components separately
- Can replace AI layer without touching compiler

**Architecture:**
```
AI Frontend (v1.1) → TypeScript → Compiler (v1.0) → ZIP
      ↑                                ↑
   Can evolve                     Frozen
```

## Supported Components (v1.1)

The AI can ONLY generate code using components supported by v1.0 compiler:

### SDK Classes
- `IFlow` - Integration Flow container
- `Component` - Generic component
- `Connection` - Connections (via `flow.connect()`)

### Component Types
- `"Enricher"` - Content Modifier only

**Example Generated Code:**
```typescript
const flow = new IFlow("MyFlow");

const modifier = new Component("CallActivity_1", "Set Body", "Enricher");
modifier.properties.bodyType = "constant";
modifier.properties.body = "Hello from AI!";

flow.addComponent(modifier);

export default flow;
```

## Limitations (v1.1 Phase 1)

### By Design (Non-Goals)

❌ **Does NOT compile TypeScript** - That's a manual step (Phase 2 will automate)  
❌ **Does NOT generate XML** - Compiler handles this  
❌ **Does NOT generate BPMN** - Compiler handles this  
❌ **Does NOT generate ZIP** - Compiler handles this  
❌ **Does NOT deploy to SAP** - Future version  

### Technical Limitations

🔲 **Limited component library** - Only Enricher (v1.0 limitation)  
🔲 **Single provider** - Only Claude (others planned for v1.2)  
🔲 **Basic validation** - No semantic analysis  
🔲 **No multi-turn conversation** - Single request/response  

**All limitations documented in docs/AI_FRONTEND.md**
